# Partner Package Outline

**Status:** L1 operational draft.  
**Purpose:** Define the materials needed to bring FQHCs, payers, KHIE, kynect/211, community organizations, and device/data partners into the program.

## Universal One-Pager

Must explain:

- Problem: rural patients have care gaps, logistics barriers, and fragmented records.
- Program: patient-owned care companion plus navigator work queue.
- First wedge: diabetic-retinopathy screening gaps.
- Safety posture: protocol-bound, navigator-supervised, no autonomous diagnosis or medication changes.
- Data posture: minimum necessary, provenance-visible, consented.
- Outcome posture: closed-loop screenings and resource connections.

## FQHC / Clinic Packet

Contents:

- Program one-pager.
- Screening workflow map.
- Site availability feed template.
- Navigator contact workflow.
- Patient enrollment script.
- Staff FAQ.
- Escalation path.
- Pilot measurement plan.

Key asks:

- Confirm screening site details.
- Confirm scheduling/contact process.
- Identify clinic champion.
- Validate patient-facing education.
- Define abnormal/referral workflow.

## Payer Packet

Contents:

- Quality-measure alignment.
- Claims data request.
- Medication/adherence future roadmap.
- MCO transportation/resource coordination use case.
- Privacy and consent posture.
- Outcomes reporting plan.

Key asks:

- Patient-authorized claims API path.
- Gap file format if API is not available.
- Pharmacy fill/adherence data plan.
- Transportation benefit workflow.

## KHIE / Data Partner Packet

Contents:

- Minimum data bundle.
- Identity matching and low-confidence queue.
- Source-provenance ledger.
- Consent scope.
- FHIR mapping direction.
- Audit and access controls.

Key asks:

- Participation pathway.
- Data elements available.
- ADT/event notification potential.
- CCD/problem/lab/screening history availability.

## kynect / 211 / Community Resource Packet

Contents:

- SDOH resource workflow.
- Resource provenance model.
- Referral vs navigator-mediated connection boundary.
- Consent/sharing workflow.
- Stale resource reporting process.
- County needs dashboard.

Key asks:

- Partner access path.
- Approved resource feed/API/export options.
- Referral creation rules.
- Resource refresh cadence.
- Community partner onboarding process.

## Device / Wearable Packet

Contents:

- Device rail roadmap.
- HealthKit/Health Connect posture.
- CGM partnership needs.
- BP cuff and smart pill bottle support model.
- Safety boundaries for insights.

Key asks:

- Supported data types.
- Patient consent flow.
- Provenance fields.
- Rate limits and data retention.

## Patient Advisory Packet

Contents:

- Plain-language product explanation.
- What Sandy can and cannot do.
- Consent and source explanation.
- Screenshots/prototype walkthrough.
- Feedback questions.

Questions to ask:

- Does the app feel trustworthy?
- Is the source/provenance language understandable?
- Are the alerts helpful or too much?
- Would you trust Sandy to help prepare questions for a navigator?
- What would make the SDOH resource flow feel respectful?
